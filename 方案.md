# 0. 方案设计

**一、总体定位 Overall Goal**
本方案目标是构建一个面向抑郁 EEG 分类与解释的多层异构知识图谱：

```text
量表回答 Scale Response
 -> 题项/选项/分数 Item/Option/Score
 -> 症状维度 Symptom Dimension
 -> UMLS 标准概念 UMLS CUI Concept
 -> MDKG 抑郁证据 Depression-specific Evidence
 -> 脑区/频段 Brain Region / EEG Band
 -> EEG 宏观状态 EEG Macro-State
 -> 分类与解释 Classification & Explanation
```

核心思想不是简单地说“量表连接 MDKG 和 UMLS”，而是：

> **CUI 负责知识融合，量表负责患者接入，症状负责语义承接，脑区/EEG 负责生理落地。**  
> **CUI aligns knowledge, scales activate patient-specific phenotypes, symptoms bridge clinical meaning, and brain/EEG layers provide physiological grounding.**

**二、数据源分工 Data Source Roles**
| 数据源       | 中文定位         | English Role                         | 主要作用                             |
| ------------ | ---------------- | ------------------------------------ | ------------------------------------ |
| 量表 JSON    | 临床观测入口     | Clinical observation anchor          | 记录题项、选项、分数、症状强度       |
| UMLS         | 概念标准化系统   | Concept normalization system         | 提供 CUI、TUI、同义词、语义类型      |
| MDKG CSV     | 抑郁领域证据网络 | Depression-specific evidence graph   | 提供疾病、症状、治疗、风险、脑区关联 |
| 脑图谱 Atlas | 脑区标准化工具   | Brain region atlas                   | AAL3 / Brainnetome / DKT ROI 映射    |
| EEG 数据     | 个体生理观测     | Individual physiological observation | 提供频段功率、连接性、宏观脑状态     |

更严谨地说：

```text
UMLS 回答：这个概念是什么？属于什么类型？

量表回答：这个患者表现出了哪些症状？强度是多少？
EEG 回答：这个患者是否存在相应脑区/频段异常？
```

**六、构建流程 Construction Pipeline**
```text
Step 1 量表层摄入
读取量表 JSON，生成 Instrument / Item / Option / Score / Domain。

Step 2 题项-症状映射
已有 CUI 直接使用；空映射通过 UMLS API、别名表、人工复核补齐。

Step 3 症状-CUI 标准化
用 UMLS CUI/TUI 判断节点类型。症状优先 T184/T033/T046，疾病优先 T047/T048。

Step 4 UMLS 子图扩展
围绕种子 CUI 做 1-2 跳扩展，保留可信 PAR/CHD/RB/RN/RO 关系。

Step 5 MDKG 实体对齐
MDKG 原始实体通过名称、别名、TUI、MDKG type 映射到 CUI。
无法映射则保留为 mdkg::* 或 local::* 节点。

Step 6 MDKG 边转译
将 help_diagnose、risk_factor_of、treatment_for、associated_with、hyponym_of 转为规范边。
associated_with 降权，measures / same_as / manifestation_of 提权。

Step 7 脑区/频段层构建
抽取 cortex、amygdala、hippocampus、ACC、DLPFC 等脑区。
无源定位时映射到 Macro_ROI；有源定位时映射到 AAL3 / Brainnetome / DKT。
建立 delta/theta/alpha/beta/gamma 及功率、连接性、同步、不对称 marker。

Step 8 图谱质控
检查每个量表是否连到症状，每个症状是否有 CUI 或 LocalConcept，每个疾病是否有足够症状证据。

Step 9 导出训练格式
导出 NetworkX、Neo4j、PyG/DGL HeteroData。
```

**七、子图生成 Subgraph Generation**
子图建议分成两类即可，不要混在一起。

```text
1. 题项模板子图 Item-template Subgraph
```

作用：描述某个量表题项在知识上通向哪些症状、疾病、脑区和 EEG marker。  
它是静态知识，不依赖具体患者。

例子：

```text
PHQ9_Q1
 -> anhedonia 快感缺失
 -> UMLS:C0002962
 -> depressive disorder 抑郁障碍
 -> reward responsiveness 奖赏反应
 -> prefrontal-striatal reward circuit 前额叶-纹状体奖赏环路
 -> frontal alpha asymmetry 额叶 alpha 不对称
```

```text
2. 个体激活子图 Subject-activated Subgraph
```

作用：根据患者选项分数，激活对应题项模板路径。  
它是动态证据，依赖具体患者。

例子：

```text
S001
 -> answered PHQ9_Q1_Option3
 -> score = 3
 -> activation_weight = 1.0
 -> activates PHQ9_Q1 template paths
```

激活权重可以定义为：

```text
activation_weight =
normalized_option_score
* item_symptom_confidence
* path_confidence
```

因此：

```text
题项模板子图 = 知识先验 Knowledge prior
个体激活子图 = 患者证据 Patient-specific evidence
```

**九、风险控制 Risk Control**
必须明确：

```text
1. 量表题项不能直接等同疾病诊断，只能测量症状维度。
2. UMLS 不是事实真理库，只负责概念标准化和类型约束。
3. MDKG 有文献挖掘噪声，associated_with 必须降权。
4. EEG 单个 0.5s/1s 切片不能直接对齐“两周症状”，必须做 macro-state pooling。
5. 没有 EEG 源定位时，脑区只能写 channel-level approximation。
6. 训练分类时应避免路径直接包含最终疾病标签造成标签泄漏。
7. 跨量表重复题目不删除，而是归一到共享 SymptomDimension。
```

**十、最终表述 Final Statement**
> 本方案以量表题项和患者选项作为临床表型观测入口，将个体症状状态映射到 UMLS CUI 标准概念，再通过 MDKG 引入抑郁领域文献证据，并进一步连接到脑区、频段和 EEG 宏观状态，形成一个可生成个体化子图、可进行跨模态对齐、可输出解释路径的多层异构知识图谱。

# 问题总结



**简单来说：你的设想在宏观逻辑上是完全成立的，但从工程和严谨的知识图谱拓扑来看，这五者的关系并不是简单的“一条直线”，而是一个以“标准概念（CUI）”为枢纽，以“量表”为锚点的多维正交架构。**

量表确实起到了极其关键的“衔接”作用，但它不是直接横亘在 MDKG 和 UMLS 中间的桥板，而是将主观的患者行为/生理（EEG）**接入**客观医学知识库（UMLS+MDKG 融合体）的“临床观测接口”。

为了让你彻底看清这套方案的底层骨架，我们对这五者的定位和关系进行结构化拆解。

## 1. 五大核心实体的系统定位

五个概念扮演着完全不同的角色，它们各司其职，共同拼接起从“宏观表型”到“微观神经生理”的完整证据链：

| **实体**             | **角色定位**       | **核心价值**                                                 | **数据属性**                               |
| -------------------- | ------------------ | ------------------------------------------------------------ | ------------------------------------------ |
| **量表 (Scale)**     | **临床观测锚点**   | 行为学操作化的工具。它是患者主观状态转化为客观数据的入口，是连接真实人类（以及伴随的 EEG）与知识图谱的起点。 | 窄而深、高度人工校准、带量化分数。         |
| **症状 (Symptom)**   | **潜在临床表型**   | 承上启下的中间态。它是量表题目所实际测量的“东西”，同时又是疾病在临床上的外在表现。 | 概念化的医疗术语，属于潜变量。             |
| **UMLS**             | **标准语义仲裁器** | 纯粹的本体词典。提供唯一的身份标识（CUI）和严格的语义分类（TUI），是融合全域知识的基础设施（通用医学规范）。 | 静态、结构化、无定量强度、定义严谨。       |
| **MDKG**             | **真实世界证据库** | 基于文献挖掘的关联网络。它告诉我们某个概念和什么药物、什么风险因子、什么生理状态有联系（专用学术支持）。 | 广而杂、带统计权重（文献支持度）、有噪声。 |
| **脑区 (Brain/EEG)** | **底层生理投射**   | 客观生物学证据。是精神疾病症状在大脑网络层面的物理/电生理映射终端。 | 解剖学空间坐标、频段、瞬时动态。           |

## 2. 它们之间是如何流转和衔接的？

理解了各自的定位，我们来看看它们是如何被缝合成一张网的。整个流转逻辑遵循“观测 -> 抽象 -> 仲裁 -> 扩展 -> 落地”的路径：

### 第一步：观测与抽象（量表 -> 症状）

患者填写了量表（例如选择了“几乎每天失眠”）。这个行为在图谱中，并不是直接连向抑郁症，而是映射到一个抽象的**症状**节点（如“失眠”）。量表是观测器，症状是被观测的表型。

### 第二步：标准化仲裁（症状 -> UMLS）

图谱中的“失眠”这个词必须有唯一的身份，否则机器无法理解。这时，**UMLS** 出场了。它给“失眠”打上了一个唯一条形码（CUI：`C0917801`），并界定它的分类（TUI：`T184 症状/体征`）。至此，患者的主观选项变成了全球通用的医学标准概念。

### 第三步：知识扩展（UMLS <-> MDKG）

这是融合最关键的一步。**真正连接 UMLS 和 MDKG 的技术桥梁，是 CUI。**

MDKG 里有海量的文献挖掘数据（例如“失眠与额叶异常相关，文献支持度 85”）。我们通过 CUI，把 MDKG 中那些带有丰富关联、带权重的科研证据，挂载到 UMLS 这个标准骨架上。

- **UMLS 负责“骨架”**（决定概念是什么、谁是谁的父类）。
- **MDKG 负责“血肉”**（提供概念之间有多大相关性、有什么治疗方案和风险因子）。

### 第四步：生理落地（扩展后的知识 -> 脑区）

顺着 MDKG 提供的科研关联路径，图谱最终指向了特定的**脑区**（如“左侧 DLPFC”）或 EEG 状态（如“Alpha 波不对称”）。这样，我们就把开头的“量表选项”与结尾的“大脑异常状态”在逻辑上贯通了。

## 3. 核心纠偏：量表到底是不是“桥”？

回到你的设想：“量表衔接起 MDKG 和 UMLS 两个图谱”。

更严谨的表述应该是：

**量表是连接“现实患者/EEG”与“双图谱融合体（UMLS+MDKG）”的临床语义桥梁。**

- **真正的技术融合桥梁是 CUI：** MDKG 和 UMLS 的融合，靠的是底层的实体对齐（把 MDKG 的自由文本映射到 UMLS 的 CUI 上）。
- **量表的作用是赋予生物学意义：** 如果没有量表，UMLS 和 MDKG 融合后，只是一个庞大但静态的医学百科全书。当把量表接入其中，这个静态百科就“活”了，它变成了一个可以输入患者实时状态（选项 + EEG），并沿着图谱路径进行动态推理的**医学辅助诊断系统**。

总结而言，你的思路方向是极为前沿且契合现代循证医学逻辑的：**用量表（行为学观测）作入口，UMLS（通用医学本体）作仲裁骨架，MDKG（文献挖掘知识）作证据网络，最终在脑区/EEG（神经生理）上完成交叉验证。** 这正是构建可解释性多模态精神疾病大模型的完美基底。



量表层 Scale/Item/Option
 -> 症状层 SymptomDimension
 -> 概念层 UMLS CUI / LocalConcept
 -> 疾病层 Disease / Disorder
 -> 脑区层 BrainRegion / EEGMarker



**量表不是把 MDKG 和 UMLS“直接衔接起来”的桥，而是把患者临床表型锚定到两个知识系统上的入口。**

### 更准确的五者关系是：

```text
量表 = 临床表型观测入口
症状 = 量表题目背后的医学/心理表型语义
UMLS = 通用医学概念标准化系统
MDKG = 抑郁领域专用知识与证据补充
脑区 = 生理机制/EEG解释层
```

所以整体关系可以写成：

```text
量表题目
-> 症状/表型维度
-> UMLS CUI 标准化
-> MDKG 抑郁领域关系增强
-> 脑区/EEG生理解释
```

或者反过来：

```text
UMLS 提供“这个概念是什么”
MDKG 提供“这个概念在抑郁研究中和什么有关”
量表提供“这个患者表现出了哪个概念”
症状是量表与医学概念之间的中间语义
脑区是从医学/症状解释走向 EEG 的生理承接层
```

因此你的表述可以改成：

> 以量表为临床表型锚点，将量表题项映射到症状/心理功能维度，再通过 UMLS 完成概念标准化，通过 MDKG 引入抑郁领域的疾病、症状、脑区及机制证据，从而融合通用医学语义与抑郁专病知识，构建面向 EEG 表征解释的多层异构知识图谱。

这比“量表衔接 MDKG 和 UMLS”更稳，因为 MDKG 和 UMLS 的融合本身主要靠实体对齐/CUI 映射；量表的真正作用是**让融合后的知识图谱能够服务于患者表型解释和 EEG 对齐**。

## 1. 用「抑郁量表」衔接 MDKG 与 UMLS —— 整体方案

### 1.1 节点设计（五层）

```
[Scale Layer]   PHQ-9, HAMD, BDI-II ...
                    │
[Item Layer]    PHQ9_Q2 "心情低落"      ← 每题一个节点
                    │

                    │
[Brain Layer]   ROI:左DLPFC, ROI:vmPFC, ROI:杏仁核   ← MDKG侧
```

### 1.2 边的类型（这是关键，不要混在一个 `related_to` 里）

| 边类型                                             | 例子                      | 来源                        |
| -------------------------------------------------- | ------------------------- | --------------------------- |
| `scale_has_item`                                   | PHQ-9 → PHQ9_Q2           | 量表说明书（人工）          |
| `item_has_option`                                  | PHQ9_Q2 → PHQ9_Q2_Opt3    | 量表说明书                  |

| `concept_isa_symptom` / `concept_manifestation_of` | C0344315 → C0011570       | **UMLS MRREL**              |
| `disease_affects_region`                           | C0011570 → 左DLPFC        | **MDKG 已有**               |

### 1.3 执行步骤

1. **建量表层**：把 PHQ-9 / HAMD-17 / BDI-II 三个表的题目和选项录入为节点（几百个，纯手工/CSV）。
2. 题目↔概念映射：
   - 自动：用 UMLS **MetaMap / QuickUMLS** 把题干文本映射到 CUI
   - 人工复核：抑郁量表只有 9~21 题，**必须人工复核**，不能纯自动（见第 3 题难点）
3. **概念→疾病/症状扩展**：调 UMLS `/CUI/{cui}/relations`，取 `RO`（other related）、`RB/RN`（broader/narrower）、`PAR/CHD`（父/子）三类关系，限定来源词表为 `SNOMEDCT_US, MSH, MEDLINEPLUS` 以免噪声。

## 3. "从量表出发"的思路评估

### 3.1 思路本身：**站得住**

**理论依据**：

- **RDoC 框架**（NIMH Research Domain Criteria）：精神疾病应该从行为/症状维度（量表是其操作化）→ 神经回路 → 分子，自上而下分层。你的设计正好契合。
- **HiTOP 模型**（Hierarchical Taxonomy of Psychopathology）：症状是底层、综合征是中层、谱系是顶层，量表是症状层的可观测代理。
- **临床现实**：抑郁的金标准诊断仍依赖量表（DSM-5 用 PHQ-9/HAMD 来操作化），EEG 是辅助证据。让量表当"语义锚"符合临床逻辑。

### 3.2 核心难点

1. 题目-概念映射的歧义
   - PHQ9_Q9 "想到自己最好死掉" 同时映射 C0038271(Suicidal ideation) 和 C0011570(MDD) 还是只前者？
   - **解法**：区分 `measures`（题目直接测的概念）和 `implies`（推断的概念），两种边权重不同。
2. 选项的语义不是离散的
   - "几乎每天" 不只是 severity=3，还隐含 chronicity。选项节点应该带**多维属性**（频率/强度/持续时间）。
3. 量表之间互不一致
   - PHQ-9 第 2 题 ≈ HAMD 第 1 题 ≈ BDI 第 1 题，但措辞不同。需要建立 `item_equivalent_to` 边（用 LLM + 专家审核）。
4. 概念-脑区的多对多
   - "Depressed mood" 涉及 vmPFC、sgACC、左 DLPFC、杏仁核……MDKG 里这些关系**强度差异巨大**，必须保留 confidence/effect_size 作为边属性，否则下游图卷积一视同仁就废了。
5. 循环论证风险
   - 如果量表标签同时用作监督信号和图谱构建依据，模型容易学到"PHQ-9 高分 → MDD 节点 → 量表高分预测"的捷径。要在**评估时用 hold-out 量表**或外部诊断标签防漏。

## 4. UMLS 中疾病和症状是怎么组织的

UMLS 不是一个本体，是**元词表（Metathesaurus）**，把 200+ 来源词表（SNOMED CT、ICD-10、MeSH、MedDRA、HPO 等）合并起来。

### 4.1 三个核心结构

| 文件/概念   | 作用                                                |
| ----------- | --------------------------------------------------- |
| **MRCONSO** | 概念→原子（atoms）：一个 CUI 包含多个不同来源的术语 |
| **MRSTY**   | 概念→语义类型（TUI）：决定它"是什么类"              |
| **MRREL**   | 概念间关系                                          |
| **MRDEF**   | 概念定义                                            |
| **MRHIER**  | 层级（来自各源词表的树）                            |

### 4.2 疾病 vs 症状的区分 —— 看 **TUI（Semantic Type）**

| TUI      | 名称                             | 含义                               | 你应该归到哪层       |
| -------- | -------------------------------- | ---------------------------------- | -------------------- |
| **T047** | Disease or Syndrome              | 疾病/综合征                        | Disease 层           |
| **T048** | Mental or Behavioral Dysfunction | 精神/行为障碍（MDD 在这里）        | Disease 层           |
| **T184** | **Sign or Symptom**              | 症状/体征                          | Symptom 层           |
| **T046** | Pathologic Function              | 病理生理过程                       | Symptom 层（可选）   |
| **T033** | Finding                          | 临床发现（覆盖更广，比 T184 模糊） | Symptom 层（弱证据） |
| T037     | Injury or Poisoning              | 损伤                               | 一般不用             |
| T191     | Neoplastic Process               | 肿瘤                               | 一般不用             |

你贴的那个 C0009044 `Closed fracture of carpal bone` 是 T037 —— 跟抑郁无关，所以 **TUI 过滤是第一步必做的事**。

### 4.3 关系类型（MRREL.REL / RELA）

- `RB / RN`：broader / narrower（同义词表内的上下位）
- `PAR / CHD`：parent / child（层级树）
- `RO`：other related（最常用，细分由 RELA 给出）
- 重要的 RELA：
  - `has_finding`、`finding_site_of`
  - `manifestation_of`、`has_manifestation` ★ 疾病↔症状的关键
  - `causative_agent_of`
  - `co_occurs_with`
  - `due_to`

**抑郁相关的最小子图抽取流程**：

1. 起点：`C0011570 (Depressive disorder)`
2. 跳 1：`manifestation_of` 反向 → 取所有症状（C0344315 抑郁心境、C0438696 快感缺失、C0917801 失眠、C0009806 注意力下降……）
3. 跳 2：每个症状再扩 1 步 `RN` / `PAR` 取细化（如失眠 → 入睡困难、早醒）
4. **过滤**：TUI ∈ {T047,T048,T184,T046}；来源 ∈ {SNOMEDCT_US, MSH, HPO}
5. 结果：~50~200 节点的"抑郁本体子图"，量级合适

## 5. c —— 严格审查

### 5.1 你的方案拆解

```
路径 P:  PHQ9_Q2 → C0344315 → C0011570 → vmPFC
        ↓ KGE (TransE / RotatE / ComplEx)
        ↓
路径向量 z_P  ──┐
                ├─ CLIP-style 对比对齐  ──┐
EEG片段 e_i ────┘                          │
↓ 编码器                                   ├─ 掩码重建（双向）
EEG向量 z_E  ─────────────────────────────┘
                ↓
        融合（add / gated）→ GCN（脑区功能图）→ 分类
```

### 5.2 严格审查（按风险从高到低）

**① "路径 KGE 向量"这个对象本身就有问题** ★ 最大风险

- TransE/RotatE 学的是**节点向量**和**关系向量**，没有原生"路径向量"。
- 三种做法：
  - (a) 路径 = 节点向量平均（信息丢失严重）
  - (b) 路径 = 各跳 `h + r` 累加（RotatE 是旋转累乘）—— 数学上正确但路径长度不一
  - (c) **用 RGCN/CompGCN 给每个节点一个 context-aware 嵌入，路径=节点序列过 Transformer**（推荐）
- **建议**：直接放弃"先 KGE 再取路径"的两步走，改成**端到端 R-GCN/HGT 编码子图 + Transformer 编码路径序列**。

**② EEG 编码器对"任意通道数和顺序"的鲁棒性**

- 直接用 CNN（输入是 `C×T` 矩阵）会**强依赖通道顺序**。
- 解法：
  - **通道作为 token，通道位置嵌入用 3D 电极坐标**（不是序号），通道间用 Transformer 自注意力 —— 这样 N、顺序都不重要。代表工作：LaBraM、EEGPT、BIOT。
  - 切分到 ROI（用源定位或简单的 10-20 → ROI 映射）后才进入对齐 —— 这能让节点对应关系更稳定，但损失了空间分辨率。
- **0.5s vs 1s**：抑郁的 alpha/theta 节律相关特征需要 ≥2s 才稳；0.5s 噪声大。建议 **1s + 50% overlap**，或者多尺度（0.5s 走精细，2s 走稳态）。

**③ CLIP-style 对齐的负样本设计**

- CLIP 负样本是 batch 内其他配对，假设是"图文一一对应"。
- 你的场景：**一个被试同时对应多条路径**（PHQ-9 9 题 × 多概念），而且**多个被试可能共享同一条路径**（都有"抑郁心境"）。
- 标准 InfoNCE 直接套**会把同病的被试当负样本互相推开**，反而破坏类内一致性。
- **解法**：用 **Supervised Contrastive Loss（SupCon）** —— 同标签的被试互为正样本，或者按"路径集合的 Jaccard 相似度"做软对比。

**④ 双向掩码重建的训练信号**

- "图谱路径 → EEG 掩码" 和 "EEG → 图谱路径掩码" 听起来对称，实际

  信息量天差地别

  ：

  - 路径 → EEG：路径信息抽象（"vmPFC 异常"），EEG 信号高维（1000+ 维），从抽象重建具体几乎不可能 → 损失会塌缩。
  - EEG → 路径：可学，因为路径是离散符号，更像分类。

- **建议**：不要硬要对称。前向（EEG → 路径）用**对比 + 路径 token 预测**；反向（路径 → EEG）退化成**路径 → ROI 异常 mask（二分类）**，而不是重建原始信号。

**⑤ GCN 卷积"对齐结果 + 功能连接图"的顺序**

- 你写的是：先对齐 → 再 GCN。
- 但功能连接图本身可能就是对齐的最佳归纳偏置 —— **应该先 GCN（用 PLV/相干性矩阵当邻接），再把节点嵌入和路径嵌入对齐**。否则对齐学到的是原始 EEG 表示，GCN 是事后装饰。

**⑥ 数据量是否撑得起**

- 抑郁 EEG 公开数据集（MODMA, HBN, TDBRAIN）合起来也就千级被试。
- 路径 + EEG + 对比学习这种自监督任务，**至少需要预训练 + 微调两阶段**。预训练用大规模健康 EEG（TUH, LaBraM 的 2500 小时），下游用抑郁数据微调。否则会严重过拟合到样本。

### 5.3 超图扩展

**为什么超图合理**：

- 普通边只能表示二元关系。但"PHQ9_Q2 + Opt3 + vmPFC异常 + alpha不对称" 是一个**四元的临床证据组**，强行拆成 6 条二元边丢信息。
- 超图就是 `hyperedge ⊆ V` 可以包多个节点。

**怎么构超图**：

| 超边类型         | 包含节点               | 例子                                     |
| ---------------- | ---------------------- | ---------------------------------------- |
| **临床证据超边** | 题目+选项+概念+疾病    | {PHQ9_Q2, Opt3, C0344315, C0011570}      |
| **生理超边**     | 多个 ROI + 频段 + 状态 | {vmPFC, 左DLPFC, theta, hyperactivation} |
| **跨模态超边** ★ | 行为节点 + 生理节点    | {PHQ9_Q2_Opt3, vmPFC↑, alpha 不对称}     |
| **时序超边**     | 同一 ROI 的连续片段    | {ROI_i@t1, ROI_i@t2, ROI_i@t3}           |

**学先后关系（你问的"超边相对顺序"）**：

- 标准 HGNN（HyperGCN, HGNN+）**没有顺序**。
- 要顺序，两个方向：
  - (a) **时序超图**：超边带时间戳，做 Temporal HGNN（参考 THGNN）。
  - (b) **序列+超图混合**：超边内部用 Transformer（attending 顺序），超边之间用 HGNN 聚合。

**有序轨迹搜索（你问的最后一点）**：

- 这本质是

  符号路径搜索 + 神经评分：

  1. 在知识超图上用 beam search 找出从"高 PHQ-9 分"出发的可能轨迹（如 `Q2_Opt3 → 抑郁心境 → vmPFC 异常 → alpha 不对称`）
  2. 每条轨迹用 GNN/Transformer 算分，分数作为类别证据
  3. 类似 **Neural-Symbolic** 或 **Path-RNN** 的范式

- 优点：**可解释**，能告诉医生"模型预测抑郁，是因为这条因果链"。

- 缺点：搜索空间爆炸，需要严格剪枝（限定跳数 ≤4，超边类型白名单）。

| **童年创伤（风险因子）** | **CTQ-SF**    | 抑郁的远端风险因子                       | 28 题    |
| **自杀风险**             | **SSRS**      | 单独建一个 high-stakes 子图              | 3 题     |

**理由**：

- **覆盖 RDoC 全部 6 域**：负性价（PHQ-9/HAMD/CTQ）、正性价（SHAPS）、唤起调节（PSQI）、认知系统（RRS）、社会过程（GAD-7 部分）、感觉运动（HAMD 精神运动迟滞条目）
- **避免重叠膨胀**：BDI/BDI-II 与 PHQ-9 高度重叠，CESD/Zung-SDS 与 HAMD 重叠，先不要
- **总计 ~129 个 item 节点 + ~520 个 option 节点**，可控
- **后期可加**：如果模型 underfit，再扩 BDI-II / CESD / IDS-SR

```python
# umls_client.py
import requests, time
from functools import lru_cache

class UMLSClient:
    BASE = "https://uts-ws.nlm.nih.gov/rest"
    VERSION = "2026AA"

    def __init__(self, api_key: str, rate_limit_sec: float = 0.05):
        self.api_key = api_key
        self.rate_limit = rate_limit_sec
        self._last = 0

    def _throttle(self):
        delta = time.time() - self._last
        if delta < self.rate_limit:
            time.sleep(self.rate_limit - delta)
        self._last = time.time()

    @lru_cache(maxsize=20000)
    def search(self, term: str, sabs: str = "SNOMEDCT_US,MSH,HPO,MEDLINEPLUS",
               search_type: str = "words", page_size: int = 10):
        """ Search a term, return top candidates. """
        self._throttle()
        url = f"{self.BASE}/search/{self.VERSION}"
        params = {
            "string": term, "apiKey": self.api_key,
            "sabs": sabs, "searchType": search_type,
            "pageSize": page_size, "returnIdType": "concept"
        }
        r = requests.get(url, params=params, timeout=20)
        r.raise_for_status()
        return r.json().get("result", {}).get("results", [])

    @lru_cache(maxsize=20000)
    def get_concept(self, cui: str):
        self._throttle()
        url = f"{self.BASE}/content/{self.VERSION}/CUI/{cui}"
        params = {"apiKey": self.api_key}
        r = requests.get(url, params=params, timeout=20)
        r.raise_for_status()
        return r.json().get("result", {})

    @lru_cache(maxsize=20000)
    def get_relations(self, cui: str, page_size: int = 100):
        self._throttle()
        url = f"{self.BASE}/content/{self.VERSION}/CUI/{cui}/relations"
        params = {"apiKey": self.api_key, "pageSize": page_size}
        r = requests.get(url, params=params, timeout=20)
        if r.status_code != 200: return []
        return r.json().get("result", [])

    def get_tuis(self, cui: str):
        c = self.get_concept(cui)
        return [st.get("name") for st in c.get("semanticTypes", [])]
```

### B.3 主映射器（多策略级联）

```python
# item_cui_mapper.py
import json, csv, re, yaml, hashlib
from pathlib import Path
from collections import defaultdict
from umls_client import UMLSClient

# --- 配置区 ---
SCALE_DIR = Path("data/ontology/reference_scales")
USE_SCALES = ["phq9", "hamd17", "madrs", "shaps", "gad7",
              "rrs", "psqi", "ctq_sf", "ssrs"]
OUT_DIR = Path("scripts/outputs"); OUT_DIR.mkdir(exist_ok=True)

# 抑郁/精神领域的语义类型白名单 (TUI)
TUI_WHITELIST = {
    "T048",  # Mental or Behavioral Dysfunction (抑郁、焦虑、PTSD 主类)
    "T047",  # Disease or Syndrome
    "T184",  # Sign or Symptom (抑郁心境、失眠、快感缺失)
    "T046",  # Pathologic Function
    "T033",  # Finding (弱证据)
    "T041",  # Mental Process (认知过程，反刍思维)
    "T055",  # Individual Behavior (自杀行为)
    "T054",  # Social Behavior
}

# 来源词表白名单：临床主流
SABS_PSYCH = "SNOMEDCT_US,MSH,HPO,MEDLINEPLUS,DSM-5,ICD10CM,NCI"

# --- 1. 已有映射收集 ---
def load_existing_mappings(scale_json: dict):
    """ 提取 JSON 中已有的 umls:: 映射"""
    out = {}
    for item in scale_json.get("items", []):
        existing = item.get("maps_to_symptoms", [])
        cuis = [s.replace("umls::", "") for s in existing if s.startswith("umls::")]
        out[item["item_id"]] = {
            "item_text": item.get("item_text", ""),
            "item_name": item.get("item_name", ""),
            "domains": item.get("maps_to_domains", []),
            "existing_cuis": cuis,
            "retrieval_terms": item.get("retrieval_terms", []),
        }
    return out

# --- 2. 人工别名表（兜底） ---
def load_aliases(path="scripts/manual_aliases.yaml"):
    """
    格式:
      "depressed_mood":
        cuis: [C0344315]
        confidence: 1.0
        rationale: "PHQ9 Q2 / HAMD Q1 通用映射，文献金标准"
    """
    if not Path(path).exists():
        return {}
    return yaml.safe_load(open(path, encoding="utf-8"))

# --- 3. 查询策略 ---
def filter_by_tui(client: UMLSClient, candidates: list, max_check=5):
    """ 取前 max_check 个候选，按 TUI 白名单过滤"""
    kept = []
    for c in candidates[:max_check]:
        cui = c.get("ui")
        if not cui or cui == "NONE":
            continue
        try:
            concept = client.get_concept(cui)
        except Exception:
            continue
        tuis_of_c = [st["uri"].rsplit("/", 1)[-1]
                     for st in concept.get("semanticTypes", [])]
        if any(t in TUI_WHITELIST for t in tuis_of_c):
            kept.append({
                "cui": cui,
                "name": c.get("name"),
                "tui": tuis_of_c,
                "root_source": c.get("rootSource"),
            })
    return kept

def query_one_term(client: UMLSClient, term: str):
    """ 多种 search type 串联"""
    cleaned = re.sub(r"\([^)]*\)", "", term).strip()  # 去括号注释
    for st in ["words", "exact", "approximate"]:
        cands = client.search(cleaned, sabs=SABS_PSYCH, search_type=st)
        if not cands: continue
        kept = filter_by_tui(client, cands)
        if kept:
            return kept, st
    return [], None

# --- 4. 主映射逻辑 ---
def map_item(client, item_id, item_info, aliases):
    """
    级联策略:
    L1: JSON 已有 umls:: → 验证（call get_concept），通过则直接用
    L2: item_name in aliases → 直接用
    L3: API search(item_text)
    L4: API search(item_name)
    L5: API search(retrieval_terms 拼接)
    L6: 全部失败 → 进 review_queue
    """
    out = {
        "item_id": item_id,
        "item_text": item_info["item_text"],
        "item_name": item_info["item_name"],
        "candidates": [],
        "strategy_used": None,
        "needs_review": True,   # default
        "confidence_auto": 0.0,
    }

    # L1
    if item_info["existing_cuis"]:
        verified = []
        for cui in item_info["existing_cuis"]:
            try:
                concept = client.get_concept(cui)
                tuis = [st["uri"].rsplit("/", 1)[-1] for st in concept.get("semanticTypes", [])]
                if any(t in TUI_WHITELIST for t in tuis):
                    verified.append({"cui": cui, "name": concept.get("name"),
                                     "tui": tuis, "root_source": "json_existing"})
            except Exception as e:
                pass
        if verified:
            out.update(candidates=verified, strategy_used="L1_json_existing",
                       needs_review=False, confidence_auto=0.95)
            return out

    # L2 别名
    if item_info["item_name"] in aliases:
        a = aliases[item_info["item_name"]]
        for cui in a["cuis"]:
            try:
                concept = client.get_concept(cui)
                out["candidates"].append({"cui": cui, "name": concept.get("name"),
                                          "tui": [st["uri"].rsplit("/", 1)[-1]
                                                  for st in concept.get("semanticTypes", [])],
                                          "root_source": "manual_alias"})
            except Exception: pass
        if out["candidates"]:
            out.update(strategy_used="L2_manual_alias",
                       needs_review=False, confidence_auto=a.get("confidence", 0.9))
            return out

    # L3 / L4 / L5
    for level, term in [("L3_item_text", item_info["item_text"]),
                        ("L4_item_name", item_info["item_name"].replace("_", " ")),
                        ("L5_retrieval", " ".join(item_info["retrieval_terms"]))]:
        if not term.strip(): continue
        cands, st_used = query_one_term(client, term)
        if cands:
            out.update(candidates=cands, strategy_used=f"{level}_{st_used}",
                       needs_review=(len(cands) > 1),    # 多候选 → 人工选
                       confidence_auto=(0.7 if len(cands) == 1 else 0.4))
            return out

    # L6 fail
    out.update(strategy_used="L6_FAIL", needs_review=True, confidence_auto=0.0)
    return out

# --- 5. 主流程 ---
def main(api_key: str):
    client = UMLSClient(api_key)
    aliases = load_aliases()

    auto_rows, review_rows = [], []
    coverage = defaultdict(lambda: {"total": 0, "L1": 0, "L2": 0,
                                     "auto": 0, "review": 0, "fail": 0})

    for sid in USE_SCALES:
        scale = json.load(open(SCALE_DIR / f"{sid}.json", encoding="utf-8"))
        items = load_existing_mappings(scale)
        for iid, info in items.items():
            res = map_item(client, iid, info, aliases)
            coverage[sid]["total"] += 1
            lvl = res["strategy_used"] or "L6_FAIL"
            if lvl.startswith("L1"): coverage[sid]["L1"] += 1
            elif lvl.startswith("L2"): coverage[sid]["L2"] += 1
            if not res["needs_review"] and lvl != "L6_FAIL":
                coverage[sid]["auto"] += 1
            elif lvl == "L6_FAIL":
                coverage[sid]["fail"] += 1
            else:
                coverage[sid]["review"] += 1

            base = {"scale_id": sid, "item_id": iid,
                    "item_text": info["item_text"],
                    "item_name": info["item_name"],
                    "strategy": lvl,
                    "confidence_auto": res["confidence_auto"]}
            for idx, c in enumerate(res["candidates"] or [{}]):
                row = {**base,
                       "cand_rank": idx,
                       "cand_cui": c.get("cui", ""),
                       "cand_name": c.get("name", ""),
                       "cand_tui": ",".join(c.get("tui", []) or []),
                       "cand_source": c.get("root_source", "")}
                if res["needs_review"] or lvl == "L6_FAIL":
                    row["reviewer_decision"] = ""    # accept/reject/replace
                    row["reviewer_cui_final"] = ""   # 复核者填
                    row["reviewer_notes"] = ""
                    review_rows.append(row)
                else:
                    auto_rows.append(row)

    # 写文件
    _write_csv(OUT_DIR / "auto_mappings.csv", auto_rows)
    _write_csv(OUT_DIR / "review_queue.csv", review_rows)
    json.dump(dict(coverage), open(OUT_DIR / "coverage_report.json", "w",
              encoding="utf-8"), indent=2, ensure_ascii=False)
    _print_coverage(coverage)

def _write_csv(path, rows):
    if not rows: return
    keys = list({k for r in rows for k in r.keys()})
    with open(path, "w", newline="", encoding="utf-8") as f:
        w = csv.DictWriter(f, fieldnames=keys); w.writeheader(); w.writerows(rows)

def _print_coverage(cov):
    print(f"{'scale':<10}{'total':>7}{'L1':>5}{'L2':>5}{'auto':>6}{'review':>8}{'fail':>6}")
    for sid, c in cov.items():
        print(f"{sid:<10}{c['total']:>7}{c['L1']:>5}{c['L2']:>5}"
              f"{c['auto']:>6}{c['review']:>8}{c['fail']:>6}")

if __name__ == "__main__":
    import sys
    main(sys.argv[1])  # python item_cui_mapper.py <UMLS_API_KEY>
```

### 1.1 MDKG 中非标准实体（如抑郁亚型）UMLS 找不到精准匹配怎么办

- **不要硬塞**：宁可保留无 CUI 状态，也不要强行匹配错的 CUI——这会污染下游所有路径

### 1.4 量表的重复/体量 与 两个图谱间的冲突 ★

### 2.1 量表之间互不一致——能否分维度去重保证一致性

**能，且应该**。用三层归一化：

→ 这样**保留差异、共享语义**，下游 R-GCN 在共享 CUI 处自然聚合，又能保留量表特定的 phrasing 信息。

### 2.2 Sequence Pooling 跨时间尺度对齐是否可行 ★

**可行，但要做对**。具体细化：

**问题本质**：

- 量表 "过去两周失眠" = 长时窗主观汇总
- EEG 0.5s 切片 = 即时电生理状态
- 直接对齐违反"语义粒度匹配"，损失会塌缩

**Macro-State 设计（具体可执行）**：

```python
# 输入: 一段 ~5min 静息 EEG, 切成 N=300 个 1s 片段, 每片 d 维
# Step 1: 编码器 f(eeg_i) → z_i ∈ R^d
# Step 2: Macro-state pooling:
#   选项 A (简单): mean(z_i) + std(z_i) concat → z_macro ∈ R^{2d}
#   选项 B (推荐): self-attention pooling, 学一个 [CLS] token 聚合
#   选项 C (序列性): 用 Mamba/Transformer 编码 z_1..z_N, 取最后 hidden
# Step 3: Macro-state 与 path embedding 做 InfoNCE
```

**为什么这能 work**：

- "过去两周失眠"在大脑留下的痕迹是**状态稳定性**（如长期 alpha 抑制），不是某个 0.5s 的瞬态
- mean+std 已经能近似捕捉静息态的稳态分布
- attention pooling 让模型学到"哪些时间点的 EEG 对'失眠'症状最有信息量"

**但有三个陷阱**：

1. **任务相关性**：静息 EEG 反映 trait（特质，匹配长时窗症状），任务 EEG 反映 state（瞬时反应）。**应只对静息态做 path-level 对齐**；任务态留给细粒度子任务（如 attention task → 对齐"注意力难集中" item）。
2. **时间窗多尺度**：不要锁死 0.5s 或 1s，应该并行：
   - 1s 切片 → 微状态/微动力学
   - 5s 切片 → 频段功率
   - 全段 macro → 与 path 对齐 多尺度向量 concat 后再做对齐
3. **路径长度归一**：path embedding 也要做 sequence pooling（不同 path 长度不同）。建议路径用 Transformer 编码后取 [CLS]，让"路径宏观语义"和"EEG 宏观状态"语义粒度对齐。

→ **结论：Sequence Pooling 可行，必须 macro-to-macro 对齐，禁止 macro-to-slice 对齐。**

### 3 关于 HGCN 学习先后顺序

你的"激活序列索引" + 位置编码的方向**思路正确**，但有更优的实现：

**三种方案对比**：

| 方案                                       | 怎么做                                                       | 优点                     | 缺点                             |
| ------------------------------------------ | ------------------------------------------------------------ | ------------------------ | -------------------------------- |
| **A. Hyperedge 内位置编码**                | 关联矩阵 H[v,e] 改为时间戳；超边内消息聚合用 Transformer attention + sinusoidal PE | 实现简单，复用 HGNN 框架 | 只能学超边内顺序，跨超边顺序丢失 |
| **B. Temporal Hypergraph (THG)** ★         | 超边有时间属性 e_t；HGCN 卷积时按时间窗滑动；模仿 TGN/TGAT   | 跨超边时序也能建模       | 实现复杂，需要时序采样           |
| **C. Directed Hyperedge + Autoregressive** | 超边节点分 "源集" / "目标集"；用 autoregressive attention 从源→目标 | 能学因果链               | 数据需要明确方向标签，监督信号弱 |

**推荐：A + B 混合**——超边内位置编码（简单）+ 全局 THG 框架（高阶时序）。

### 4 关于"路径表示用 R-GCN + Transformer"的合理性 ★

你问这点很关键，深入分析。

**为什么不直接用 KGE (TransE/RotatE)**：

| 缺陷               | TransE/RotatE                      | R-GCN+Transformer                           |
| ------------------ | ---------------------------------- | ------------------------------------------- |
| **关系类型多样性** | 关系当独立向量，10+ 种关系训不出来 | R-GCN 每种关系独立线性变换，天然处理多关系  |
| **节点上下文**     | 节点嵌入与邻居无关                 | GCN 邻居聚合 → 上下文感知                   |
| **路径=节点序列**  | 没有"路径"概念，只能事后拼接       | Transformer 编码序列，[CLS] 直接给路径向量  |
| **异构性**         | 必须人工设计哪些算 entity          | R-GCN/HGT 原生支持异构                      |
| **可扩展性**       | 增加新节点要重训整个 KGE           | GCN 类用 inductive learning 直接 generalize |

**R-GCN+Transformer 的具体架构**：

```
Step 1 [节点编码]
  每个节点 v 有初始特征 x_v:
    - cui 节点: PubMedBERT(name + def) → 768d
    - item 节点: BioBERT(item_text) → 768d
    - roi 节点: 一组 hand-crafted features (MNI 坐标, 网络归属) + Embed → 64d
    - option 节点: one-hot(score) + linear → 64d
  → 投影到统一维度 d_h
   ⇣
Step 2 [R-GCN 多层异构传播]
  for layer in 1..L:
    h_v^{l+1} = σ( Σ_{r∈R} Σ_{u∈N_r(v)} (1/c_{v,r}) W_r^l h_u^l + W_0^l h_v^l )
  R = {scale_has_item, item_measures_symptom, symptom_manifestation_of, ...}
   ⇣
Step 3 [路径采样] (训练时随机, 推理时 beam search)
  path = [v_1, r_1, v_2, r_2, ..., v_k]
  典型 path: item → cui_symptom → cui_disease → roi
   ⇣
Step 4 [Transformer 编码路径]
  token_i = h_{v_i} (来自 R-GCN) + RelEmb(r_i) + PE(i)
  在路径前加 [CLS]
  z_path = Transformer(tokens)[CLS]
   ⇣
Step 5 [跨模态对齐]
  与 EEG macro-state z_macro 做 SupCon Loss
```

**R-GCN 比 GAT/GCN 优势**：

- 多关系类型显式建模（不像 GAT 把所有关系混在 attention 里）
- 参数有上界，关系数 R 不爆炸
- 与 UMLS-style 异构图天然契合

**Transformer 比 LSTM/Path-RNN 优势**：

- 长路径下不衰减
- 路径中"哪一跳贡献最大"可解释（attention weight）
- 双向上下文（path 没有强方向时）

**可改进点（你后续可加）**：

- 用 **HGT (Heterogeneous Graph Transformer)** 替代 R-GCN：节点/关系类型都进 attention，表达更强
- 用 **CompGCN**：把关系嵌入与节点嵌入组合（h ⊗ r），参数比 R-GCN 还少

→ **最终结论**：先 R-GCN+Transformer 跑通基线，跑通后升级 HGT，路径有序时 Transformer 加 causal mask。

## 一、AAL3 / Brainnetome 是什么 —— 脑区"标准化"的含义

### 1.1 为什么需要标准化标签

MDKG 里的 anatomy 实体长这样（我马上从你的 CSV 里抽几条给你看）：

看到了吗？同一个脑区被写成**七八种不同字符串**：

- `prefrontal cortex` / `dorsolateral prefrontal cortex` / `orbitofrontal cortex` —— 不同层次
- `amygdala` (在 signs / physiology / region 三种 type 下都出现)
- `hippocampal formation` / `CA1 field of hippocampus` / `hippocampal volumes` —— 解剖 vs 形态学度量混在一起
- `frontal lobe` / `left frontal lobe` / `right frontal lobe` —— 左右拆分不统一

**580 个 unique 解剖串** 但实际可能只对应 ~50 个真正的脑区。这就是为什么需要"标准化"。

### 1.2 AAL3 / Brainnetome / DKT —— 三个常用脑图谱

"脑图谱（brain atlas）"= **把整个大脑划分成 N 个标号脑区**的标准方案。每个方案是一个 NIfTI 文件 + 一份 ID→区名 LUT。EEG 圈、fMRI 圈互相参照同一份图谱才能比对。

| 图谱                             | 区数        | 划分方式                             | 优缺点                                                       | EEG 适配性                                |
| -------------------------------- | ----------- | ------------------------------------ | ------------------------------------------------------------ | ----------------------------------------- |
| **AAL** (2002) / **AAL3** (2020) | 116 / 166   | 解剖学手工分区（皮层 + 皮层下）      | 最广用、好对照文献；AAL3 补了细分丘脑、苍白球、海马回，更适合情绪研究 | 一般，皮层区粗                            |
| **Brainnetome (BNA)**            | 246         | 解剖 + 连接组学半自动分区            | 细，每区附带功能连接 profile，适合做"脑区-行为"对齐          | 较好，作者本身做了 EEG 投射版本           |
| **Desikan-Killiany (DKT)**       | 68（皮层）  | FreeSurfer 表面分区                  | 与 sMRI/EEG source 估计无缝                                  | **最好**，多数 EEG 源定位 pipeline 默认它 |
| **Schaefer-100/200/400**         | 100/200/400 | 功能分区 + 7 Network/17 Network 标签 | 与功能网络（DMN/CEN/SN）直接对应                             | 良好                                      |
| **Yeo-7 / Yeo-17**               | 7 / 17      | 仅大网络                             | 超粗，适合做高级解释                                         | 良好                                      |

### 1.3 EEG 端"对应"是什么意思 —— **从电极到脑区的映射**

EEG 直接测的是**头皮电极**（如 64 通道 10-10 系统）。要和上述脑图谱对齐，有三档精度：

**【档 1：粗糙映射 / 不做源定位】** —— 最常用

```python
# 简单按"通道在哪片头皮"对应一个粗解剖区
CHANNEL_TO_REGION = {
  "Fp1/Fp2/AF3/AF4/F3/F4/Fz": "Prefrontal_Cortex (PFC)",
  "F7/F8/FC5/FC6":            "Inferior_Frontal",
  "C3/C4/Cz":                 "Sensorimotor",
  "T7/T8":                    "Temporal",
  "P3/P4/Pz/CP5/CP6":         "Parietal",
  "O1/O2/Oz":                 "Occipital",
}
```

**优点**：零依赖，快。**缺点**：颅外电极 ≠ 颅内来源（容积传导，一个电极信号是多脑区混合），分辨率只有 ~6-8 区。

**【档 2：sLORETA / eLORETA / MNE / beamformer 源定位】** —— 严肃 EEG 研究的主流

1. 把每个通道的电压通过**逆问题**反推到大脑容积（grid 上数千个 voxel 的 source activity）
2. 把 voxel 用 AAL3/BNA/DKT 之类的图谱聚合成脑区 ROI
3. 输出 `(N_subjects, N_ROIs, T)` 而不是 `(N_subjects, N_channels, T)`
4. 现成工具：MNE-Python 的 `mne.beamformer.lcmv` / Brainstorm / FieldTrip
5. **需要个体 MRI 或 fsaverage 模板头模型**

**【档 3：source-localized + atlas-mapped 频段功率】** —— 最匹配你的 KG 设计

- 对每个 ROI、每个频段（delta/theta/alpha/beta/gamma）算功率/连接性
- 输出 `(N_subjects, N_ROIs, N_bands, N_features)` —— **这正好对应你 KG 里 `EEG_Feature` 节点的 `preferred_channels` 和 `band` 属性**

### 1.4 那"标准化到 AAL3"具体做什么

UMLS 不是一个单一疾病本体，而是把很多医学词表统一到 Metathesaurus。核心单位是：

```
CUI: concept identifier，概念 AUI: atom identifier，某来源词表中的一个术语原子 SUI: string identifier，字符串 MRCONSO.RRF: 概念名、来源、同义词 MRSTY.RRF: semantic type MRREL.RRF: 关系 MRDEF.RRF: 定义 MRSAT.RRF: 属性
```

疾病/症状并不是两个干净的表，而是靠 semantic type、source vocabulary 和 relation 共同判断。常用类型包括：

```
Disease or Syndrome Mental or Behavioral Dysfunction Sign or Symptom Finding Pathologic Function Clinical Attribute Individual Behavior Mental Process Anatomical Structure Body Location or Region
```

要注意：UMLS CUI 层的 /relations 可以查相关关系，但 NLM 明确说明不在 CUI 层断言 parent/child；父子层级主要来自 source-asserted vocabularies，比如 SNOMED CT、MeSH、ICD、LOINC 等。因此你要查“疾病层级/症状层级”，应优先用 source-level ancestors/parents/relations，而不是只看 CUI relations。

官方依据：UMLS 是多词表集成系统， 不是一个单一疾病本体，而是由 Metathesaurus、Semantic Network、SPECIALIST Lexicon 三部分组成。Metathesaurus 用 CUI 聚合同义概念，Semantic Network 给每个 CUI 分配 semantic type。NLM 文档说明 Metathesaurus 按概念组织，所有概念至少有一个 semantic type；Semantic Network 提供概念大类和类型间关系。2026AA 约有 3.53M concepts 和 18.06M unique concept names。

对你最重要的是这些类型：

```text
症状/体征：
Finding
Sign or Symptom

疾病/病理：
Pathologic Function
Disease or Syndrome
Mental or Behavioral Dysfunction

脑区/解剖：
Body Part, Organ, or Organ Component
Body Location or Region

功能/过程：
Mental Process
Physiologic Function

量表/工具：
Diagnostic Procedure / Intellectual Product / Clinical Attribute 等，需按来源确认
```

UMLS 当前语义类型页把 `Finding -> Sign or Symptom` 放在 conceptual entity 下，把 `Pathologic Function -> Disease or Syndrome -> Mental or Behavioral Dysfunction` 放在 biologic/pathologic function 下。

API 层面，UMLS search 已支持用 `semanticTypes` 或 `semanticGroups` 限制结果，例如只搜 Disorders 或只搜疾病/病理相关类型；每次 API 调用需要 `apiKey`。([documentation.uts.nlm.nih.gov](https://documentation.uts.nlm.nih.gov/rest/search/)) 但要注意，UMLS concept relations 多为 source-asserted，NLM 文档也提醒 concept 层关系不应被直接理解成 NLM 自己断言的父子疾病层级。([documentation.uts.nlm.nih.gov](https://documentation.uts.nlm.nih.gov/rest/relations/index.html?utm_source=openai))
